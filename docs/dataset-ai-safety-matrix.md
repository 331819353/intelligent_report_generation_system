# 数据集 AI 修改安全校验矩阵（2026-07-22 至 2026-07-23）

## 目标与口径

本轮以已发布的“客户每月支付金额详情”画布为固定基线，使用真实 `deepseek-v3` 逐条提交修改需求。结果分为：正常通过、合理澄清、正确拒绝、错误拦截和错误接受。测试只应用到浏览器内存画布；预览完成后撤销，未保存数据集。

基线包含 3 个数据节点、1 个日期转换、2 个 INNER JOIN、1 个输出分组、3 个维度、1 个指标和 4 个输出字段。

## 修复策略

- 保留最终的精确 `current → plan` 差异校验，不允许通过降低安全规则换取通过率。
- 将已校验的 `fieldChanges` 确定性落到选列、分组用途和最终输出；规划器遗漏数组修改时不再误杀，未授权的其他变化仍由最终校验拒绝。
- `purpose` 由最终用途数组派生，消除 `SELECTED_ONLY` 与实际分组用途自相矛盾。
- 对唯一物理血缘的转换绑定，收敛 `transform_N` 和 `_trimmed` 派生名；冲突或多候选继续失败关闭。
- 日期、文本、数值、空值和条件等处理均由同一语义意图模型生成结构化 `TRANSFORM` 操作，不再为客户名称、月份或订单数建立原句特判。
- 实体数量根据授权表的业务名、标签、语义类型、主标识和完整血缘选择计数字段；支付/退款表中的同名外键不会仅因词面相同而抢占订单实体。
- 关联能力边界由语义意图和结构化 Schema 判断；不支持的类型进入业务化澄清，不能静默降级为 LEFT/INNER，也不在模型调用前扫描原句关键词。
- 名称、说明和任意其他配置都由 `UPDATE.fields` 精确授权；通用补丁执行器只合并点名字段，不需要“只改名称”专用后处理。

## 2026-07-23 通用语义编辑回归

“不再关注月份了”只作为派生语义删除的一个样例，本轮没有增加月份专用规则。意图输入新增由可信当前图生成的 `derivedFields` 目录，统一提供任意转换产物的组件类型、中英文名称、稳定编码、物理血缘、精确引用位置和直接消费者。用户按业务语义唯一指向维度、指标或处理结果时，模型必须直接生成完整变更，不再要求组件名称或 ID；只有多个业务候选无法由上下游角色区分时才澄清。

服务端仍只接受资产目录中的真实物理字段。模型把现有转换产物 ID 或 code 写入 `fieldChanges` 时，会通过当前图唯一血缘归一为 `nodeId + tableId + column`；歧义绑定保持失败关闭。若已校验的最终字段用途证明某个转换的全部产物均无引用，服务端会确定性补全该转换的删除和每个直接消费者的旁路改线。该规则由图结构触发，适用于日期、文本、数值及其他转换组件，不解析“月份”关键词。MODIFY 方案先校验响应外壳，再落实锁定的字段数组、转换删除和改线，随后重新执行完整图校验与精确 `current → plan` 差异校验，避免可确定修复还未执行就提前误判。

| 真实自然语言需求 | 候选结果 | 应用后结构 | 结束节点预览 |
|---|---|---|---|
| 不再关注月份了 | 自动删除 `DATE_FORMAT`，`join_1.left` 从 `transform_1` 改接 `node_1`，删除月份维度和输出 | 0 个字段处理、2 个维度、2 个指标、4 个输出 | 5 行；`customer_id`、`customer_name`、`monthly_paid_amount`、`order_count` |

两项均只应用到浏览器内存画布，预览成功后执行撤销；最终恢复为 1 个字段处理、3 个维度、2 个指标和 5 个输出，未保存。单元回归另覆盖日期派生维度、`TEXT_TRIM` 文本派生维度和 `NUMBER_ROUNDING` 数值派生指标的血缘归一、引用删除、组件清理、消费者改线，以及“先锁定补全、后完整图校验”的安全顺序。

## 2026-07-23 统一补丁执行与开放语料回归

新增、更换、移除、换位和配置修改仅作为回归矩阵的五个结构维度，运行时没有五路分类器。所有修改请求，包括多动作组合、英文、含糊业务表达和不支持能力，都先进入 `dataset-dag-intent-v12`；`READY` 进入同一个 changeSet 补丁执行器，确有多个同等合理目标时返回可回答的 `CLARIFY`。

| 覆盖维度 | 自动化断言 |
|---|---|
| 新增 | 新组件必须有精确下游锚点，产物被 Group/End 真实消费；孤立新增在意图阶段修复或拒绝 |
| 更换 | 同类更新保留稳定 ID；跨类型使用 REMOVE + ADD + 消费者改线，字段口径不丢失 |
| 移除 | 删除组件并旁路所有直接消费者，未涉及组件从当前画布恢复 |
| 换位 | 只改变数据流拓扑，移动组件保持身份和配置，前驱、后继同时改线 |
| 配置修改 | 只合并 `UPDATE.fields` 点名值，模型顺手改名、改线或添加组件会被确定性剔除 |
| 组合与开放表达 | 同一句可包含多个动作；英文、隐含目标和普通业务描述均进入语义模型，不走关键词捷径 |
| 模糊与非法能力 | 唯一目标直接修改；真正歧义返回业务问题；不支持能力不被静默替换 |

真实 `deepseek-v3` 使用完全相同的原始要求“在现有流程中新增一个数值取整组件，把实付金额四舍五入保留 0 位小数后用于分组汇总和最终输出，其余逻辑不变”完成复测。意图阶段 22 秒成功，规划器首版 139 秒后自动修复一次，最终返回 `join_2 → transform_2(NUMBER_ROUNDING) → group_1 → end_1`；分组指标和最终输出都引用 `transform_2.rounded_paid_amount`，原 `DATE_FORMAT`、两次 INNER JOIN、客户维度和订单数保持不变。该验证只生成提案，没有应用、保存或发布数据集。
