# 统一报告渲染器

前端正式渲染入口位于 `web/src/components/report/ReportRenderer.tsx`。设计器预览和在线查看器不得自行拼装报告页面，必须向该入口传入版本化报告 JSON 和本次运行上下文。

## 渲染链路

```text
报告 JSON V1
  -> Draft 2020-12 Schema 校验
  -> 页面/分块二维网格
  -> 组件注册表
  -> 单组件错误边界
```

- `ValidatedReportRenderer` 校验不可信 JSON；除未知组件类型外，任何合同错误都会阻止整页渲染并显示精确路径。
- `ReportRenderer` 只依赖已校验的 `ReportDocument`、`ReportRuntimeContext` 和渲染模式。
- `defaultComponentRegistry` 按组件类型选择渲染函数。未知类型进入安全占位视图，不执行来源 JSON 中的脚本或 HTML。
- `ReportErrorBoundary` 将渲染异常限制在当前组件网格内，其他组件继续展示。

## 运行上下文

运行上下文与报告定义分离，当前包含：

- 报告参数值；
- 按组件 ID 索引的 `READY/LOADING/ERROR` 数据状态；
- 权限编码和角色编码快照。

分块和组件的 `requiredPermission`、`allowedRoleCodes` 都会参与前端可见性判断。前端判断仅用于界面收口，服务端仍必须在查询和运行数据返回前完成真实授权，不能依赖隐藏 DOM 实现安全隔离。

## 设计器与查看器一致性

`ViewerPage` 与尚未持久化的新建入口使用同一份 `api/examples/report-json-v1.json` 兼容样例；已有 UUID 的 `DesignerPage` 改为加载服务端规范草稿。两种模式继续复用同一个渲染入口和运行上下文形状，自动化测试比较示例设计态与查看态的 `data-component-id` 集合，确保组件树一致；模式差异只允许影响设计辅助线、分块标签和编辑交互。

## 长画板与布局编辑

- 主画板使用 1920×1080、12×10 的逻辑首屏，纵向高度按页面所有分块的最大 `y+h` 动态增长。
- 容器窄于 1920 px 时只缩小画板，不放大也不产生横向滚动；逻辑行高和导出基准保持不变。
- 设计器支持指针拖拽、右下角手柄缩放、方向键移动和 `Shift + 方向键` 缩放。所有坐标先吸附为整数网格，再执行 12 列边界、碰撞和最大编辑行数校验。
- 布局更新采用不可变文档副本；已有 UUID 报告从服务端规范草稿加载，`minimumRowsByPage` 作为独立 `editorState` 保存，不进入正式报告 JSON。旧 `report-layout-draft:{reportId}` / `report-designer-session-v1` 只在一次性迁移中读取，绝不再作为草稿事实源。

## 服务端草稿保存边界（T0409 核心已实现）

- 持久化后的报告以服务端 UUID 统一数据库主键、对象权限 ID 和 `report.id`，`report.code` 只作为租户内业务编码。草稿必须保持 `report.status=DRAFT`；前端不能通过完整保存或 Patch 直接生成发布/归档状态。
- 正常保存携带 `expectedRevision`、`Idempotency-Key`、有序语义 `changes[].patch` 和客户端结果完整 JSON。服务端从数据库旧草稿逐 change 重放并重新规范化，最终哈希不一致、版本过期、越权、占用或任一操作失败时整批回滚。
- Patch 仅允许 `add/remove/replace`，拒绝根替换、`test/move/copy/from`；严格校验 RFC 6901 转义、数组下标和末尾 `-`。请求信封、规范化前后完整 JSON、Patch 字节数、100 条操作上限和路径长度分别受 T0409 的硬上限约束，显式旧草稿恢复也不能绕过身份、旧文档锁和完整合同校验。
- 服务端读取、修订查询、正常保存及幂等重放都按当前可信操作者重新授权；更新事务还使用变更前草稿检查 `REPORT/UPDATE`、分块布局/配置/数据快照锁、组件 `manualLocked`、活跃 `report_edit_guards` 和乐观锁版本。在同一批中先解锁再修改原受保护内容仍然失败，`report:edit` 只决定当前 UI 是否显示编辑控件。
- 指针移动和缩放从按下到释放合并为一个语义操作、一个撤销历史项和一批 Patch；移动过程只更新本地预览，取消手势不保存。每份草稿只允许一个在途请求，后续编辑排入下一批；旧响应只能确认其覆盖的客户端修订，不能清除请求发出后产生的脏状态。
- 设计器展示 `CLEAN/DIRTY/SAVING/CONFLICT/ERROR`；未持久化的新建模板从打开起即为 `DIRTY`。当前已覆盖普通站内链接和整页卸载确认，SPA 前进/后退及非链接路由阻断仍在 TODO；不能把“请求已发送”误当作“服务端已确认”。
- 旧会话迁移先加载服务端草稿，再解析同一报告键下的 `report-designer-session-v1` 信封或更早期裸 V1 文档。本地报告身份、Schema、完整 JSON 或 `minimumRowsByPage` 不合法时不得恢复；本地与服务端规范哈希相同则清除旧键，不同则提示存在差异并允许导出，禁止自动上传。用户确认后以当前服务端版本执行完整恢复，只有保存成功才能删除旧键。
- 刷新或 409 冲突重载后清空本地撤销/重做栈，不从旧 Patch 日志无条件重建。补偿操作必须引用同报告的原始非补偿修订、保持稳定 target 一致、通过正式编辑语义，并让整份规范哈希精确回到被引用修订的 before/after；多人实时协同和 CRDT 不属于 T0409。
- 草稿完整 JSON、派生组件/依赖索引、语义修订、幂等响应和通用审计在同一租户事务提交。修订中的 tenant、actor、时间、来源和 Patch 哈希由服务端确定，组件复制生成新 ID 并记录原件/副本；首次创建前的本地编辑聚合进 `REPORT_CREATE`，创建后每个定义变化才有独立客户端操作 ID。通用 `audit_logs` 摘要不重复保存 Patch 正文、结论正文或运行数据值。

真正的 `report_version`、固定数据集/指标依赖、发布任务与编辑租约、reports 桶不可覆盖 JSON 制品、哈希/大小校验和当前已发布指针均由 T0601 完成。T0409 草稿保存成功不等于发布成功，也不能产生在线查看器可访问的半发布制品。

## 浏览态冻结

- `report-sticky.ts` 在逻辑坐标中生成冻结计划；配置的 `sticky.top` 是视口 CSS 像素，进入计算前除以当前画布缩放比，因此 1、0.5 等缩放下的可见顶部留白一致。
- 分块支持页面范围或显式所属页面容器；组件支持所属分块、页面，或显式所属页面/分块容器。`containerId` 只与报告实体 ID 比较，从不拼接为 DOM 选择器；未知、非祖先及页面/分块同名歧义全部失败关闭。
- 计划器按自然网格位置稳定遍历，只让实际可见、有权限且已挂载的项参与；横向区间相交的同范围项才纵向堆叠，容量不足时相关项恢复普通文档流。父分块和子组件同时冻结时，子项位移会扣除父项已产生的位移。
- 共享渲染器只在查看态消费冻结计划。它定位真实纵向滚动祖先，使用 `requestAnimationFrame` 合并滚动和尺寸变化，并对页面作用域组件解除所属分块裁剪；设计态保持原始网格，避免冻结干扰拖拽。冻结层级和位移只在元素真正离开自然位置后生效，待命项仍使用普通布局层级；子组件层级不会提升整个所属分块。
- V1 保留 CSS 原生层叠组边界：父分块显式配置普通 `zIndex` 或父分块自身发生冻结位移时，子组件的冻结层级只在该父组内比较；没有父层叠组时，页面作用域子组件可在画布根层级参与比较。若产品后续要求子组件无条件跨父组全局排序，需要在在线查看器中引入保持单一 DOM/焦点顺序的页面级提升层。
- 查看态始终为冻结规划保留等高状态槽；降级告警在槽内按普通文档流展示，既不会因出现/消失改变 `canvasTop` 并反向触发规划结果振荡，也不会覆盖正文或键盘焦点。
- 设计器点击或聚焦分块/组件后，可编辑开关、作用域、祖先容器、顶部偏移和层级。所有更新尊重分块配置锁，禁用时规范化为 `{"enabled":false}`，并进入统一正/逆 Patch 撤销历史与服务端待保存队列。
- 窄屏、触控、打印和 PDF 媒体样式强制清除位移、层级和宿主溢出，恢复普通文档流，避免正文和焦点被遮挡。

## 分块内动态网格

- 每个 `M×N` 主网格分块派生 `4M×4N` 内网格，组件坐标只使用内网格整数值。
- 组件可从左侧面板原生拖入；键盘和触屏用户也可先选择组件类型，再点击分块空白处插入。拖放类型会重新收敛到报告合同枚举，不直接信任浏览器传入字符串。
- 组件支持指针移动、缩放手柄、方向键移动、`Shift + 方向键` 缩放、`Ctrl/Cmd + D` 复制和 `Delete/Backspace` 删除。组件事件会阻止冒泡，避免误触发外层分块移动。
- 分块缩放时按新尺寸重新生成内网格，并按比例调整组件后确定性寻找最近空位；无法容纳全部组件时拒绝修改并返回中文路径错误。
- `report-component-catalog.ts` 是首批组件的唯一目录，集中声明中文名称、类别、默认配置、默认尺寸和最小尺寸；设计器面板和组件工厂均从目录派生。
- 组件缩放和分块缩放都会校验正式最小尺寸。默认尺寸会按目标分块收缩，最小尺寸不超过空白 `1×1` 分块对应的 `4×4` 内网格，因此从空白基础单元新增任一首批组件仍然成立。

## 第一批正式组件

- 内容类：标题、结构化富文本、附加信息卡；富文本使用段落、标题、引用和列表项等结构化块，不执行来源 HTML。
- 数据类：筛选器、指标卡、表格、柱状图、条形图、折线图、饼图和结论。
- 媒体与元信息类：图片、附件、数据来源和更新时间。
- 数据型组件统一读取 `READY/LOADING/ERROR` 运行状态；异常抛给单组件错误边界，空数据使用统一空态。
- 图片和附件只接受站内相对地址或受支持的资源协议，拒绝 `javascript:` 等可执行协议。正式对象存储签名、域名白名单与 CSP 仍由 T0802 收口。
- 表格当前最多渲染 20 行作为首批安全上限；完整分页、虚拟化和图表大数据抽样分别归入 T0602、T0803。

报告 JSON V1 通过按组件类型的条件 Schema 收紧 `style`、`binding`、`interaction` 和 `refreshPolicy`，拒绝未知配置键、非法图表类型及小于最小尺寸的网格。Go 校验器独立执行关键配置、最小尺寸和结论引用语义校验，不能依赖浏览器校验结果。

## 筛选、联动和数据绑定

- 报告 JSON 中的 FILTER 和图表 linkage 始终通过稳定的 `parameterId` 引用参数定义；运行上下文和运行命令始终以 `Parameter.code` 为键保存参数值。前后端纯解析器负责唯一一次 ID → code 转换，组件不能把 ID 当成运行参数名，也不能由展示名称猜测参数。
- 参数的 `semanticBinding` 明确声明报告语义字段，并为每个目标数据集版本保存唯一的 `datasetVersionId + fieldId + datasetParameterCode` 映射。同一个报告参数可映射到两个独立数据集的不同字段和参数编码；缺失、重复或不完整的映射全部失败关闭，不按同名字段自动兜底。
- `effectScope.kind` 支持 `REPORT`、`PAGE`、`BLOCK` 和 `COMPONENTS`。报告、页面和分块作用域按文档位置确定性派生目标；指定组件作用域只接受显式且不重复的组件 ID。页面参数既不能由其他页面触发，也不能影响其他页面。
- 前端 `report-interactions.ts` 与后端 `internal/reportbinding` 执行同一组核心规则：定位交互来源、解析参数、重算作用域、选择精确语义映射并输出确定性的组件目标。当前 Go 解析器是无 IO 的领域边界；精确读取已发布数据集版本并执行 `ONLINE` 查询由 T0601/T0602 接入，浏览器传来的目标列表不能直接成为服务端执行依据。
- 图表运行数据可提供机器可读点 `points[]`，每个点包含稳定 `id`、展示 `label`、数值 `value` 和 `semanticValues`。点击联动只读取与参数语义字段对应的机器值，不使用标签反解析业务值；来源字段还必须是图表 `dimensions` 中已声明的维度。查看态以可聚焦按钮暴露可联动点，SVG 仅承担图形展示。
- 图表下钻由 `interaction.drill.levels` 声明 `fieldId`、`semanticFieldCode` 和中文标签；运行态只保存当前层级，支持逐级下钻和返回上级，不把临时层级写回报告 JSON。
- 每次参数或图表交互都会分配递增 `requestRevision`。异步执行结果按受影响组件采用 latest-wins：重叠目标只接受各自最新代次，不相交目标可独立完成，避免旧结果覆盖新结果或把无关组件永久留在加载态。多目标查询的主动取消、部分失败、重试和服务端乱序控制仍由 T0602 完成。
- 校验分为两层：草稿加载时由 Draft 2020-12 Schema 与 Go 报告校验器检查结构、参数引用、语义映射、作用域和静态目标；交互发生时由前后端解析器重新计算实际目标，对重复标识、未知组件、页面越界、缺失/歧义映射、操作符基数和非法点击维度失败关闭。运行错误只进入交互状态或可重试的组件错误态，不修改版本化报告定义。

真实浏览器已验证月份筛选和图表数据点点击都会同步更新参数、图表与依赖结论，季度→月份下钻及两级返回可恢复汇总状态；交互按钮可获得键盘焦点，选中点暴露 `aria-pressed`，1280 px 视口无横向溢出且控制台无运行错误。完整键盘、移动端和快速连续点击矩阵仍按 T0801 执行。

## 分块重置与撤销历史

- 清空和删除都会从正式报告文档释放目标分块，但在历史操作类型中保留不同语义；锁定分块不能执行两种操作。
- 非空分块先显示页面内确认对话框，明确待移除组件数量。对话框支持初始焦点、Tab 循环和 Escape 取消，不使用会阻塞浏览器线程的原生 `window.confirm`。
- 未占用主网格由 `deriveEmptyGridCells` 表达为可选择的隐式 `1×1` 单元，不写入报告 JSON。选中单元或将组件放入其中时，才创建正式 `1×1` 分块。
- 分块删除后若正式文档高度收缩，设计器会在独立 `editorState` 中保留页面最小行数，并随服务端草稿保存，确保原区域刷新后仍可选择；查看器继续只读取合同中的正式动态高度。
- 统一历史栈覆盖分块和组件的新增、移动、缩放、复制、删除及分块重置，支持按钮和常用快捷键，最多保留 100 项；只保留一份当前完整 JSON，past/future 保存正逆 Patch。刷新或显式远端重载会清空内存历史，服务端以追加式不可变修订保存已确认语义操作，修订查询接口单次有界。
- `report:edit` 与服务端返回的 `capabilities.edit` 只负责前端只读降级。保存事务重新校验 `REPORT/UPDATE`、变更前文档锁、组件人工锁、`report_edit_guards` 占用和版本冲突；T0601 的发布任务负责创建、续租及释放占用。
